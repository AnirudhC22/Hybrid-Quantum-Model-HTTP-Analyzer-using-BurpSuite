# Thesis Fact-Check Report

Cross-referencing all claims in `thesis.pdf` against the actual source code in `d:\web_attack_detector\`.

---

## 🔴 Critical Issues (Factually Wrong)

### 1. Framework Mismatch: Flask vs FastAPI
| Thesis Claims | Actual Code |
|---|---|
| "FastAPI backend" | `api_server.py` uses **Flask**, not FastAPI |

The thesis mentions FastAPI, but [api_server.py](file:///d:/web_attack_detector/api_server.py) clearly imports and uses Flask:
```python
from flask import Flask, request, jsonify, send_from_directory
app = Flask(__name__)
```

> [!CAUTION]
> This is a direct factual error. The entire API is built on Flask, not FastAPI.

---

### 2. Architecture Mismatch: Feature Pipeline (TF-IDF + PCA → Wrong)
| Thesis Claims | Actual Code |
|---|---|
| "Classical Preprocessing: **TF-IDF + PCA**" | Features are **30 hand-crafted features** (entropy, keyword counts, char ratios), NOT TF-IDF |
| "100 features → 64 → 4 via 2-layer neural network" | No neural network for feature reduction. PCA reduces 30 features directly to 4 |

The thesis describes a `TF-IDF vectorization → PCA` pipeline, but [feature_extractor.py](file:///d:/web_attack_detector/utils/feature_extractor.py) does **none of that**. It manually computes 30 domain-specific features:
- Length features, entropy, special character counts, keyword matches for SQLi/XSS/CMDi/traversal, URL-encoded characters, structure signals, and ratio features.

There is **no TF-IDF anywhere in the codebase**. The thesis claims "100 features down to 4" but the actual code has exactly **30 features** reduced to 4 via PCA in the quantum module.

> [!CAUTION]
> The entire feature extraction pipeline described in the thesis is wrong. The actual system uses 30 hand-crafted features, NOT TF-IDF vectorization of raw HTTP payloads.

### 3. Neural Network Dimensionality Reduction — Does Not Exist
| Thesis Claims | Actual Code |
|---|---|
| "Phase 1: A 2-layer neural network (100 → 64 → 4)" | No neural network exists for this purpose |

The thesis describes a classical neural network compressing features before the quantum circuit. In reality, [quantum_ml.py](file:///d:/web_attack_detector/models/quantum_ml.py) uses `sklearn.preprocessing.StandardScaler` + `sklearn.decomposition.PCA(n_components=4)` — basic linear projection, not a learned neural network.

### 5. Dataset Claims — CSIC 2010 Not Actually Used
| Thesis Claims | Actual Code |
|---|---|
| "Uses the **CSIC 2010 HTTP Dataset**" | Uses a **synthetic dataset** generated by `dataset_generator.py` |
| "Contains thousands of benign and malicious requests" | Default: 600 samples per class × 5 classes = **3,000 total** |

The thesis references the CSIC 2010 dataset as the benchmark, but [dataset_generator.py](file:///d:/web_attack_detector/data/dataset_generator.py) generates **entirely synthetic data** from hardcoded payload templates. The CSIC 2010 dataset is never loaded, parsed, or referenced anywhere in the code.

> [!CAUTION]
> The thesis claims results on an established benchmark dataset (CSIC 2010) but the system actually trains on **synthetic data** with only ~3,000 samples by default. This fundamentally undermines the reproducibility and comparability of the reported results.

### 6. Massive Inflation of Dataset Size
| Thesis Claims (Visualizations) | Actual Code |
|---|---|
| Total dataset: **425,000 samples** (318,750 normal + 106,250 malicious) | Default: **3,000 samples** (600 per class × 5 classes) |
| "28,350 SQLi, 24,120 XSS, 15,940 CMDi, 13,830 Path Traversal..." | 600 per class, equally balanced |

The `Research_Paper_Visualizations.py` (Figure 7) claims dataset sizes of 425,000 total samples. The actual training code defaults to 600 per class. Even the maximum `--samples` argument is typically set to 800.

> [!CAUTION]
> The claimed dataset size in the thesis (~425K samples) is roughly **140× larger** than what the code actually trains on (~3K samples). This is a massive factual discrepancy.

### 8. Closed-Ring Entanglement Claim — Not in Code
| Thesis Claims | Actual Code |
|---|---|
| "**Closed-Ring Entanglement** (CNOT gates from qubit 3 back to 0)" | Only linear CNOT chain: 0→1, 1→2, 2→3. **No wrap-around.** |

The thesis specifically highlights a "closed ring" entanglement pattern (CNOT from qubit 3 → qubit 0). The actual circuit in `quantum_ml.py`:
```python
for i in range(N_QUBITS - 1):    # 0,1,2 only
    qml.CNOT(wires=[i, i + 1])   # 0→1, 1→2, 2→3
```
There is **no CNOT from qubit 3 back to qubit 0**. This is a linear chain, not a ring.

### 9. Measurement — Single Qubit vs All Qubits
| Thesis Claims | Actual Code |
|---|---|
| "Pauli-Z expectation values are calculated for **each qubit**" | Only `qml.expval(qml.PauliZ(0))` — measures **qubit 0 only** |

The thesis says all 4 qubits are measured. The code only measures qubit 0:
```python
return qml.expval(qml.PauliZ(0))  # Only qubit 0!
```

The OvR (One-vs-Rest) strategy trains a separate circuit per class and takes the single PauliZ(0) output from each — not 4 measurements from one circuit.
